## Files

- `bootstrap_venv.py`: creates `.venv` and installs dependencies from `requirements.txt`.
- `cicd_prepare.py`: prepares `source-workspace.yaml` and `target-workspace.yaml` from `context` or explicit CLI parameters.
- `demo/cicd_seed.py`: creates versionable artifacts and sample resources in the source workspace for controlled demos and tests.
- `cicd_publish_source.py`: recreates the stage bundle from the source workspace and versions it in Git.
- `cicd_publish_target.py`: consumes the Git content in the target workspace, synchronizes the deploy bundle, deploys it, and reconciles names.
- `demo/cicd_destroy.py`: fully removes the `source` and `target` workspaces used only for demos and controlled transport-flow tests.
- `cicd_deploy.py`: bundle deploy executor for the target workspace.
- `cicd-orchestrator.ini`: centralizes the orchestrator's customizable operational parameters.

## Architecture Direction

The project is organized around two use cases:

- `source publish`: prepares `bundle_stage` in the source workspace and publishes its content to Git.
- `target publish`: consumes content from Git, synchronizes the deploy `bundle/` in the target workspace, deploys it, and reconciles resource names.

This structure keeps shared logic in internal modules while exposing separate scripts only where the use case actually changes.

## Safe Versioning

Local files with real environment values should not be committed to Git, for example:

- `source-workspace.yaml`
- `target-workspace.yaml`
- `commands.sh`
- `logs/`
- the local `.downloads/` cache containing the latest `.whl` downloaded from the official release

The files `source-workspace.yaml` and `target-workspace.yaml` are generated by the scripts based on `cicd-orchestrator.ini`, `context`, and user-provided parameters. They should be treated as local environment artifacts, not as versioned example files.

Script execution examples should live in this `README.md`.

## Authentication

The scripts support these `--auth-method` values:

- `api_key`
- `instance_principal`
- `resource_principal`
- `oke_workload_identity`

When `--auth-method` is not provided, the default comes from `cicd-orchestrator.ini`.

Typical usage:

- `api_key`: local execution, personal notebook, or simple automation outside OCI.
- `instance_principal`: execution inside an OCI VM with proper IAM policies.
- `resource_principal`: execution in OCI services that expose resource principal.
- `oke_workload_identity`: execution inside OKE workloads with workload identity configured.

Quick examples:

```bash
python cicd_publish_target.py --auth-method api_key
python cicd_publish_target.py --auth-method instance_principal
python cicd_publish_target.py --auth-method resource_principal
python cicd_publish_target.py --auth-method oke_workload_identity
```

## Usage Flows

### Real production flow

In the real production process, the orchestrator should not depend on the source workspace. The developer is responsible for:

1. preparing the content in the source workspace
2. generating the bundle to be transported
3. publishing that content to the Git repository
4. opening the merge request that triggers the pipeline

Inside the pipeline, the process should interact only with the target workspace:

```text
Developer workspace
    |
    |  manual bundle generation + Git publish
    v
Git repository
    |
    |  CI/CD pipeline
    v
Target workspace
    |
    `-> sync deploy bundle -> deploy -> reconcile resource names
```

Expected operational flow:

```text
prepare target
    -> publish target
```

Script mapping:

```text
cicd_prepare.py target
    -> cicd_publish_target.py
```

### End-to-end demo flow

The demo flow exists to validate the full process end to end, including the part that, in production, would remain the developer's responsibility in the source workspace.

It reuses the same target-side publish step used by the real flow:

```text
demo destroy
    -> prepare source
    -> prepare target
    -> demo seed
    -> publish source
    -> publish target
```

Visually:

```text
demo source workspace
    |
    |  demo/cicd_seed.py
    v
publish source
    |
    |  creates bundle_stage + publishes to Git
    v
Git repository
    |
    |  same target-side release logic used in production
    v
publish target
    |
    `-> sync deploy bundle -> deploy -> reconcile resource names
```

Script mapping:

```text
demo/cicd_destroy.py
    -> cicd_prepare.py source
    -> cicd_prepare.py target
    -> demo/cicd_seed.py
    -> cicd_publish_source.py
    -> cicd_publish_target.py
```

### End-to-end demo example

```bash
python demo/cicd_destroy.py --auth-method api_key --yes

python cicd_prepare.py --auth-method api_key source \
  --region us-ashburn-1 \
  --aidp-ocid <AIDP_OCID> \
  --workspace-name developer \
  --repository-url <REPO_URL> \
  --credential-name <AIDP_GIT_CREDENTIAL_NAME>

python cicd_prepare.py --auth-method api_key target \
  --region us-ashburn-1 \
  --aidp-ocid <AIDP_OCID> \
  --workspace-name production \
  --repository-url <REPO_URL> \
  --credential-name <AIDP_GIT_CREDENTIAL_NAME>

python demo/cicd_seed.py --auth-method api_key

python cicd_publish_source.py \
  --auth-method api_key \
  --commit-message "chore: publish developer workspace transport bundle"

python cicd_publish_target.py \
  --auth-method api_key \
  --commit-message "chore: publish transport bundle into production workspace"
```

Prepare the source workspace:

```bash
python cicd_prepare.py --auth-method api_key source \
  --region us-ashburn-1 \
  --aidp-ocid <AIDP_OCID> \
  --workspace-name developer \
  --repository-url <REPO_URL> \
  --branch main \
  --parent-dir /Workspace/cicd \
  --bundle-name bundle \
  --credential-name <AIDP_GIT_CREDENTIAL_NAME>
```

Prepare the target workspace:

```bash
python cicd_prepare.py --auth-method api_key target \
  --region us-ashburn-1 \
  --aidp-ocid <AIDP_OCID> \
  --workspace-name production \
  --repository-url <REPO_URL> \
  --branch main \
  --parent-dir /Workspace/cicd \
  --bundle-name bundle \
  --stage-bundle-name bundle_stage \
  --credential-name <AIDP_GIT_CREDENTIAL_NAME>
```

When `--context <name>` is used, target-side `prepare` reads all required static values from `contexts/<name>.yaml`. It does not depend on `source-workspace*.yaml` as an input anymore.

## Demo Seed

```bash
python demo/cicd_seed.py --auth-method api_key
```

The seed creates:

- notebooks under `src/`
- a file under `shared/`
- one real cluster in the workspace
- two real jobs in the workspace, pointing to the created notebooks

Important:

- the seed does not create `bundle/` as a regular folder
- the `bundle/` path must be materialized by `cicd_publish_target.py` as a bundle resource
- if `bundle/` already exists as a regular folder in the workspace, remove it before validating the flow

## Publish Source

By default, the script uses `./source-workspace.yaml` and `./target-workspace.yaml`.
Shared operational parameters live in `./cicd-orchestrator.ini`.

Configuration precedence:

- CLI argument
- workspace YAML (`source-workspace.yaml` or `target-workspace.yaml`)
- `cicd-orchestrator.ini`
- internal code fallback

For `cicd_prepare.py`, the recommended source of declarative configuration is the selected `context` file. The generated workspace YAML files should be treated as resolved runtime state, not as the primary source of truth.

```bash
python cicd_publish_source.py \
  --auth-method api_key \
  --commit-message "chore: publish developer workspace transport bundle"
```

## Publish Target

The target-side publish consumes the latest Git-versioned content to complete deploy and reconciliation.

```bash
python cicd_publish_target.py \
  --auth-method api_key \
  --commit-message "chore: publish transport bundle into production workspace"
```

## Prepare Target

On the target side, `prepare` guarantees only the workspace, the base directory, and the Git folder.
Creating or recreating the deploy bundle shell in `bundle/` is now the responsibility of `cicd_publish_target.py`, which handles this resource idempotently whenever it needs to publish and deploy.
Target-side `prepare` must get its required static values from `context` or explicit CLI arguments. It must not depend on `source-workspace*.yaml` as an input.

## Destroy

Fully removes the `source` and `target` workspaces defined in the YAML files generated by `prepare`.
This script is exclusive to demos and controlled tests; it is not part of the normal CI/CD production flow:

```bash
python demo/cicd_destroy.py --auth-method api_key --yes
```

Without `--yes`, the script shows the workspace names and keys that will be removed and asks for interactive confirmation.
With `--yes`, it shows the same warning and waits 5 seconds before starting the destruction.

To destroy only one side:

```bash
python demo/cicd_destroy.py --auth-method api_key --yes --skip-target
python demo/cicd_destroy.py --auth-method api_key --yes --skip-source
```

## Notes

- The stage transport bundle lives in `bundle_stage/` inside the Git folder.
- The canonical deploy bundle lives in `bundle/`.
- On the source side, `cicd_publish_source.py` always recreates the same `bundle_stage/`; if a previous version exists, it is removed before the new one is created.
- Git must always transport `bundle_stage/`; the target-side `bundle/` is kept as the deploy bundle.
- On the target side, the script preserves the `bundle/` root, the `.aidp` folder, and tolerates a local `.gitignore`; it removes only the remaining internal content and copies the content from `bundle_stage/` into it without overwriting the local `.aidp`.
- `cicd_publish_target.py` ensures the deploy bundle shell on the target when needed; `prepare target` does not need to materialize that resource in advance.
- Folder classification should not rely only on `type/resourceType`: the script uses Git metadata for `git_folder` and the presence of `aidp_workbench.yaml` and `.aidp/resource_origins.yaml` for `bundle_folder`.
- To inspect the raw payload and computed classification for a path, use `python tools/cicd_inspect_object.py --auth-method api_key --config target-workspace.yaml --path /Workspace/...`.
- In the seed flow, the cluster may end in `ACTIVE` or `STOPPED`; `STOPPED` is enough to keep the resource transportable and referenciable by jobs.
- If manual console validation requires running notebooks or jobs immediately, start the cluster manually when it is in `STOPPED`.
- The `bundle/` path must not be pre-created by seed or manually as a regular folder, otherwise AIDP may preserve the destination as a `Folder` instead of a `Bundle`.
- The commit/push from `cicd_publish_source.py` is executed through the AIDP Git API (`commitPush`), not through local `git` on the host.
- Bundle deploy uses the `dataLakes/20240831/.../bundles/actions/deploy` endpoint, which was manually validated through the console.
- If the remote repository is empty, the flow is still valid, but the source side must have its Git folder created first by `cicd_prepare.py`.
- Log retention and other operational defaults are centralized in [cicd-orchestrator.ini](/home/inicoli/TRAMPO/OCI/AIDP/oci-aidp-cicd-orchestrator/cicd-orchestrator.ini); loading and fallback logic for those defaults lives in [core/settings.py](/home/inicoli/TRAMPO/OCI/AIDP/oci-aidp-cicd-orchestrator/core/settings.py).
- The supported authentication methods are `api_key`, `instance_principal`, `resource_principal`, and `oke_workload_identity`; provide `--auth-method` explicitly when needed.
